这里在删除句子时首先通过 memset 将 puVar1[2] 区域，也就是节点中的 char* sentence_ptr指向的地址，置零。需要注意，这里的 if 语句判定 *sentence_ptr 第一字节不为0才会进行打印操作，但是 free 操作在置零之后，而在执行 free 时会将 fd 写入用户区，这意味着即使这些数据被清零，仍可能使检查通过：

在低版本 glibc (如 2.23)，fd 由 free 直接写入（高版本加入指针加密），此处虽然通过 memset 将指针指向地址数据清空，但节点依然存在，由 GLIBC 保存在 fastbin 链表中，也就是节点依然通过 fd 相连。

于是句子缓冲区(堆)被挂载到 fastbin，而因为 free((void *)puVar1[2]) 后并未将 puVar1[2] 置零，节点中的指针字段依然指向这块内存，一旦这片内存被再次使用，使用之前的节点便可导致此处内存泄露。这就是 Use After Free（UAF）的标准使用方式。

二、解决方案

泄露 libc

基于以上信息，我们形成以下思路：

  1. 索引一个40字节的句子(选项2)（因为40字节刚好是后面节点的大小，我们让两者大小相同才可通过 fastbin 复用）。
  2. 删除句子(选项1)。
  3. 此时我们在 fastbin 中存在一个40字节的空闲块(旧缓冲区)，但所有旧节点依然存在，所有 sentence_ptr 依然指向空闲块，也就是旧的句子缓冲区。
  4. 索引一个比原句子大16字节以上的新句子（我们需要让新节点malloc(0x28)复用 fastbin 中已释放内存，而不是句子缓冲区复用内存，这里为了防止内部碎片管理机制把已释放内存分配给句子缓冲区，索引一个比原句子大16字节以上的新句子更稳妥）。
  5. 此时分配了一个新句子缓冲区，而 fastbin 中的空闲块被复用，作为新的节点。注意这里原链表依然存在，新节点会继续连接到旧链表。虽然新节点的 sentence_ptr 字段会更新为新缓冲区地址，但是旧节点的 sentence_ptr 依然指向旧缓冲区，也就是此时的新节点。
  6. 查找旧节点中的单词，然后删除句子。我们现在在fastbin中有一个40字节的空闲块，而这个块仍然被用作节点，这是第二次释放此处内存。
  7. 索引一个40字节的句子，再次复用此处内存，控制节点内容，写入 got 表项。
  8. 当我们搜索该节点，伪造指针指向的内容会被打印出来，这会导致内存泄漏。

Fastbin Double Free

所以现在我们有了 libc 地址，接下来思路就是将 onegadget 或 system 写入 free_hook。

具体 _int_free 的 double free 检查如下：

当要释放一个 fastbin 大小的堆块 p 时，系统会先找到对应的 fastbin 链表，并用 old 变量记录下当前位于链表头部的第一个空闲块，之后执行一个检查 old == p，也就是判断正在释放的堆块 (p) 是否和链表最前面的空闲块 (old) 是同一个；如果 old == p 条件成立，意味着用户试图连续两次释放同一个堆块，此时程序就会因 “double free or corruption (fasttop)” 错误而终止。

我们发现，double free 只检查链表头部第一个空闲块，如果我们释放 chunk1 后，再释放 chunk2 ，这样 main_arena 就指向 chunk2 而不是 chunk1 了，此时我们再去释放 chunk1 就不再会被检测到。另外，链表会形成环。以下为示例程序：

以上便是 Double Free 技术，那么这个技术到底实现了什么呢？下面详细探究：

复用 fastbin 中的空闲块时，glibc 堆管理器会把 main_arena 指向的堆块进行分配，然后将 main_arena 更新为这个堆的 fd 指针。

当你调用 index 创建一个新句子时，程序会 malloc 一块内存，并把你的输入写入这块内存。如果我们能控制 malloc 返回的地址是 __free_hook，那么我们输入的句子内容就会直接覆盖掉 __free_hook 的值。Fastbin Double Free 的作用，就是欺骗 glibc，让它误以为 libc 数据段上的某个地址是一个空闲的 fastbin 块。逻辑链条是这样的：

  1. Double Free：我们让同一块 40 字节的堆内存被释放两次。这会让 fastbin 链表形成一个环（chunk A -> chunk B -> chunk A ->...）。

  2. 链表劫持：因为链表里有重复的地址，我们可以通过第一次 malloc 拿到 chunk A，向 chunk A 里写入数据。由于 chunk A 同时也是链表的第三个节点，我们写入的数据就覆盖了 fastbin 链表的 fd 指针。

  3. 指向任意地址：我们把 fd 指针修改为 __free_hook 附近的地址。这样，glibc 就会认为那个地址是下一个空闲块。

  4. 分配到目标：第二次 malloc(40) 返回 chunk B。第三次 malloc(40) 返回 chunk A。 第四次 malloc(40)就会返回我们伪造的那个地址（也就是 __free_hook 附近）。

  5. hook 劫持：此时输入的句子内容就会直接覆盖掉 __free_hook 的值。

字节错位 & 劫持hook

所以我们有了chunk2->chunk3->chunk2的环，现在只需要向__malloc_hook写入onegadget 地址。我们的思路是通过修改某个空闲块的 fd 指针使其指向 __malloc_hook，再次复用空闲块时 glibc 就会认为 __malloc_hook地址为某个堆地址，于是我们就可以向其中写入数据。

这里补充 malloc 复用空闲块的逻辑，简单来说 malloc 会根据申请大小查找 fastbin，如果存在对应空闲块就把 main_arena 存储的值返回，然后把 main_arena 更新为此空闲块的 fd 指针。需要注意这里的先后顺序：在 fd 指针写入到 main_arena 后才会有数据输入到复用的空闲块，我们需要根据这点理解脚本。

实际上，这是因为 malloc 在复用fastbin空闲块时，会有一个 size 字段的校验，这个 size 字段的检查正是在 _int_malloc 函数内部实现的，它是 malloc 核心分配逻辑的一部分。简单来说，假如 malloc 复用一个0x70的空闲块，它会检查元数据部分 size 字段是否为 0x70，如果不是则证明这不是一个有效的块。很明显 __malloc_hook区域几乎不可能出现完全符合堆块结构的内存。

字节错位技巧则完美解决了这个问题。观察上面的调试信息，在__malloc_hook上方存在一些数据，要构成字节错位，我们让地址偏移某个距离，使得size字段(fd+0x8)刚好变成0x70或fastbin接受的其他数值，来通过检查。查看字节存储或许更清晰：

可以看到，高位的 0x70 会存储在高字节，这时我们构造一个偏移让 fd+0x8 落在 <_IO_wide_data_0+304> 这一段的 0x7a 处，于是size字段就是 0x7a（0x70 0x00 0x00 0x00 0x00 0x00 0x00 0x00），这就是字节错位技巧。

另外，由于ASRL，这里数据的高位在0x70~0x7f之间变化，而标准的 Fastbin 大小只有 0x20, 0x30, 0x40, 0x50, 0x60, 0x70, 0x80。没有 0x7a 或 0x7b 这种大小。

那 0x7a 为什么能用来欺骗系统呢？这其实涉及到 安全检查的精度的问题：

  • 分配时的校验逻辑：当 glibc 从 Fastbin 取出一个 chunk 时，它会计算 chunksize()，并通过 fastbin_index() 宏来确定这个大小属于哪个 bin。关键在于，这个索引的计算是右移 4 位 (即除以 16) 然后减 2 。
  • 计算结果是宽泛的区间：无论是 0x78、0x79、0x7a… 一直到 0x7f，右移 4 位后的结果都与 0x70 相同。
    • 公式：idx = (sz >> 4) - 2
    • 代入 0x7d：(0x7d >> 4) - 2 = 7 - 2 = 5（对应 0x70 的 bin）
    • 代入 0x70：(0x70 >> 4) - 2 = 7 - 2 = 5
  • 所以，对于负责“验票”的 fastbin_index() 宏来说，0x7a 和 0x70 指向的是同一个入口。因此，即使伪造的 size 是 0x7a，glibc 也会把它当作 0x70 这个 bin 里的合法成员，从而让检查通过。这就是为什么字节错位伪造的 0x7a 能够绕过硬件的 size 校验。

需要注意，不要使用__malloc_hook地址上方8字节数据去错位，因为 glibc 会认为 fastbin 返回的指针是堆的元数据，之后会向用户区读入数据(+0x10偏移)，如果太靠近__malloc_hook，__malloc_hook会有一部分作为size字段，另一部分作为用户区，会导致 one-gadget 地址被截断。通常来说使用远一点的偏移再通过填充覆写__malloc_hook比较可靠。

需要注意填充19字节而非35字节，因为有16字节作为了元数据。

构造ROP链

所以我们有了 chunk2->chunk3->chunk2 环，思路和上面劫持 hook 相似，接下来就是构造字节错位去修改栈（返回地址）。我们要找到 ret 时栈的状态，尝试构造字节错位，这里贴出调试过程：

此时我已经有了：[+] Stack leak: 0x7fff60140460 ，单步执行到某个 ret 指令处，查看栈状态：

接下来的 ret 指令会将栈顶数据弹入rip(返回地址)， 然后跳转到 rip 。

需要注意，这里有多个 ret 指令，但我们不能直接使用它们，原因很简单，我们无法通过调试观察 index 写入操作后栈的状态，这里所有的 ret 都在写入操作之前，我们读入数据修改的内存根本不会被当作返回地址，甚至我们连 size 字段的检查都无法通过（栈在不断变化），这个尝试也提醒我们不要完全相信栈上的数据。

那么这里如何劫持返回地址呢？我参考的解决方案都没有详细说明exp里的偏移是如何得到的，不过这里我有一个想法，我们在 pwndbg 调试时经常可以看到 [BACKTRACE] 一栏，这是 main 函数返回、析构函数处理完毕后的 exit 退出流，即 _start -> __libc_start_main -> exit 的调用链：

也就是说，调用 exit 后，程序的执行流会沿着这个调用栈逐步返回，最终通过 _exit 系统调用终止进程。而这里的__libc_start_main+240，只有在调用 exit 后才会被 backtrace。所以我们可以修改这里的数据，然后再调用 exit 让程序 backtrace 到我们的ROP链即可。这样就不用担心读入数据前栈被改变。

注意这里的回溯其实都在栈上，我们依然是修改栈上内容，比如这里我们就可以看到__libc_start_main+240所在的位置：

查看附近数据构造字节错位：

我们可以利用 0x7ffc77287098 这一行，此处存储的是 0x0000000000400890，这个地址的高位字节 0x40 是静态的，这使得它成为伪造 chunk 大小时的一个可靠数值。并且高位字节 0x40 之前存在8字节 0x00，于是我们可以伪造 size 字段为 0x40。(这里我又忘记把 double_free函数实现中申请堆块的大小改为0x30，注意一下)

另外，fastbin 链表允许的大小最大为0x80，所以我们最多读取 0x80 字节，如果使用一个较小的字节错位需要注意是否越界（比如构造字节错位 0x7f，我们的ROP链加上填充垃圾数据长度必须要小于 0x70，这里用0x40满足）。

小注：这里在栈上调试构造字节错位时最好去 malloc 附近找，因为此时的栈布局基本不变。我调试时发现经过一个puts 后原先构造的字节错位就被覆盖了。
