1）随机数

首先解释__fd = open("/dev/urandom",0);，它直接调用了 Linux 的系统调用 open，具体参数是：

  • 路径：/dev/urandom
  • 标志：0 (即 O_RDONLY，只读模式)

最终它会返回一个文件描述符（非负整数）存入 __fd。

这里又引出了两个问题：

  1. /dev/urandom是什么：

    要理解 /dev/urandom，关键在于理解 Linux 的一个重要设计哲学：”一切皆文件”。它看起来像一个文件，但实际上它连接了内核和用户。

    /dev/urandom 并不是一个存储在硬盘上的数据文件，它是一个字符设备文件（Character Device File），作为内核随机数生成器的接口 。虽然它是一个特殊的设备文件，但它拥有和普通文件完全相同的接口。内核中 /dev/urandom 设备的驱动程序实现了 open、read、close 等标准系统调用。

    它的工作流程可以理解为一个熵池（Entropy Pool）系统：

    • 收集”环境噪音”：内核会持续不断地收集来自硬件和系统的各种微小扰动作为”种子”。这些来源包括键盘敲击的时间间隔、鼠标的移动轨迹、磁盘读写的时序、硬件中断信号等。这些看似杂乱无章、无法预测的信息被称为”熵”（Entropy）。
    • 混合与生成：这些熵被收集进一个叫”熵池”的容器中，并通过一个密码学安全的伪随机数生成器（CSPRNG），将它们”搅拌”并生成高质量的随机数 。
    • 像文件一样读取：当你的程序调用 read(__fd, ...) 时，内核会拦截这个请求，转而从 CSPRNG 中取出一串符合要求的随机数，复制到程序内存中。

    这种方式极大提高了安全性，如果说 rand() 生成的伪随机数可以被预测，那么这里的伪随机数几乎无法被预测。详细可以看这里

  2. 返回的文件描述符是什么：

    你可能注意到一些函数会用到文件描述符，比如 read(0,buf,size) 就是从标准输入读取。在 Linux 内核中，每个进程都有一个文件描述符表（file descriptor table）：

    注意，0、1、2已经被标准输入输出占用，所以 open 返回的第一个文件描述符通常是 3。

    所以，之后sVar2 = read(__fd,&local_28,0x10);是从 /dev/urandom 读入0x10字节，更准确地说，这会触发内核随机数生成器，生成指定数量的随机字节，拷贝到内存。

2）地址计算

接下来需要解释这里：

目的：在内存的高地址区域（PIE 程序所在的区域）随机选一个位置，用来 mmap 一块新内存作为自定义堆。

  • local_28 是从 /dev/urandom 读来的 64 位随机数
  • 直接用它当地址不行——64 位地址空间太大，大部分区域要么是内核空间，要么是没映射的空洞
  • 取模 0x555555543000 的作用是：把随机数“裁剪”到一个合法的、可用的地址范围内（大约在 0 到 0x555555543000 之间）
  • 然后再 + 0x10000 和 & ~0xFFF（页对齐），得到一个真正可以 mmap 的随机页地址

没有这个除法：mmap 会失败，或者映射到一个根本不能用的地址。这个地址作为基址。最终计算出的地址落在0x10000 到 0x555555553000 之间。

之后程序使用 mmap 分配内存：

参数含义：

  • __addr：希望映射的起始地址。
  • 0x1000：映射长度，即 4096 字节（正好一页）。
  • 3：内存保护标志 PROT_READ | PROT_WRITE。这页内存可读可写。
  • 0x22：映射类型标志，转换为二进制就是10 0010 (MAP_PRIVATE | MAP_ANONYMOUS)。
    • MAP_PRIVATE：私有映射，修改此内存不影响其他进程。
    • MAP_ANONYMOUS：匿名映射，不关联任何文件，纯粹分配了一段内存（类似 malloc 背后做的事）。
    • 在这种情况下，__addr 只是个建议值。内核会尽量在这个地址分配，但如果该地址已被占用，内核会自动在附近找一个空闲地址，并返回那个新地址。如果被占用就直接失败，但都不会强制覆盖。
  • -1：文件描述符，告诉内核，这块内存不需要和任何磁盘文件关联。我们只是想”捏造”一块内存来用。
  • 0：偏移量，既然没有文件，也就不存在从文件的哪个位置开始映射的问题，所以偏移量必须是 0

(已知 MUL 后的 RDX 就是乘法结果的高 64 位，右移了多大、这个魔数怎么来的，取决于除数和数据宽度。对于这里的 64 位无符号除以 29：魔数 ≈ 2^64 / 29 ≈ 0x8d3dcb08d3dcb0d)

所以我们知道了这里就是让随机数 local_20 对 0xe80 取余数，那么为什么除数偏要选择 0xe80 呢？前面讲解 mmap 参数时讲到它映射了 0x1000 字节 (4096) 大小内存，此处 0xe80 (3712) 比这个值稍小，意味着偏移量只能在 0 到 0xe7f (0~3711) 之间，这就保证 基址 + 偏移 绝不会超出 4096 字节的保护页边界，杜绝了堆越过 mmap 边界引发段错误。

总结一下整个流程的完整意义：程序从内核取16字节的随机数，其中后8字节计算出随机基址，前8字节计算出随机偏移，最终返回 基址 + 偏移。

上面分析了一下这个程序有意思的地方，比较明显的漏洞是 Fill 选项这里：

此处输入大小由我们指定，并没有结束条件，明显构成堆溢出。

另外需要注意，这里堆块分配是由 calloc 完成的（参数void *calloc(size_t nmemb, size_t size);），与 malloc 不同，calloc 的特性是会将所分配内存全部清零，它只做三件事：

  1. 从系统堆申请指定大小的内存
  2. 把这块内存全部清零
  3. 返回这块内存的地址

calloc 同 malloc 一样也有 chunk header，此处堆块信息还被程序记录到了上面说的随即分配的地址处。

三、解决方案

可以确定的是，我们主要的漏洞就是任意长度堆溢出。由于随即地址存储堆块信息，我们似乎无法得到并修改堆指针；另外程序开启了PIE，这也是需要解决的问题。

思路

  • 通过 unsorted bin 泄露 libc 的 main_arena 地址
  • 利用 fastbin attack 任意地址写

泄露 libc

程序的 Dump 选项提供输出功能，我们需要释放一个堆块使其进入 unsorted bin ，之后通过 Dump 输出此处的 fd 和 bk 完成 libc 泄露。但是，释放一个堆块后程序会改变堆块自定义的标志(为0)，这里需要解决程序 if 条件检查：

分析程序可知，堆块自定义的标志存储在之前讲到的随即地址，另外还有PIE的限制，我们基本无法直接修改这里的标志来绕过 if 条件检查。

接下来一个自然的思路就是利用程序本身逻辑去修改标志：比如，先构造一个 fastbin 链，再释放一个堆块使其进入 unsorted bin，然后让这个堆块连接到 fastbin 并再次分配，同时让标志变成 1。这个思路和下面的利用方案类似，但是这个思路最大的问题就是程序使用 calloc 分配堆块，而 calloc 会清零申请的内存，原来的 fd 和 bk 会被覆写。这里如果使用 malloc 这个思路应该可行。

利用思路：

  1. 申请 4 个0x10堆块、1个0x80堆块，记为 chunk0~4， 先后释放 chunk2 和 chunk1，构造fastbin 链表。
  2. Fill 选项编辑 chunk0，通过堆溢出修改单字节，使 chunk1 的 fd 指针指向 chunk4，即将 chunk4 加入 fastbin。main_arena->chunk1->chunk4->0
  3. 编辑 chunk3，通过堆溢出修改 chunk4 的 size域 为0x21，以满足检查。
  4. 再次申请两个0x10堆块，会复用 chunk1 和 chunk4。特别注意，这里 chunk4 又使用了 idx = 2 的槽位，同时 chunk4 未被释放，还占用原来的 idx = 4 槽位。
  5. 释放 idx = 4 (chunk4，idx=4的标志变为0)，使其进入unsorted bin，然后 Dump 选项输出 idx = 2 (还是chunk4，idx=2的标志仍然是1)
