Posted on 2025-12-08 Edited on 2025-12-14 In Writeup

onlyfgets

一、程序分析

程序只有NX保护，且有简单栈溢出。但是，查看程序没有system等可以直接获得shell的gadget

有signal和alarm，根据提示gadget：

注：

endbr64（End Branch 64-bit）是 Intel CET（Control-flow Enforcement Technology）技术的一部分，用于防止ROP攻击。

作用：endbr64 ; 标记这是一个合法的间接跳转目标

工作原理：

  1. 运行时检查：当程序通过间接跳转（如 call rax, jmp [rbx]）执行时
  2. 如果目标地址不是 endbr64 指令 → 触发异常
  3. 如果是 endbr64 指令→ 正常执行

分析一下两个原语

假设我们可以控制 rbp 和 ebx，就可以向任意地址 (rbp - 0x3d) 加任意值 (ebx)

二、调试过程

（一） 原语利用

step1

根据上述分析，我们尝试控制 rbp 和 ebx，查一下 pop 指令：

可以看到，我们有了gadget

  • 0x000000000040114d : pop rbp ; ret
  • 0x00000000004011fc : pop rdi ; ret

但是无法直接控制 ebx。思考，ebx 可以通过 bl + dh 修改，通过调试命令查看一下dh

注：

  1. 输入

    • -e：启用转义字符解释

    • AAAAAAAA...：40个 ‘A’（32缓存，8 rbp）

    • \xae\x10\x40\x00\x00\x00\x00\x00：小端序的 0x4010ae（add bl, dh gadget）

  2. 断点：

    • -ex：执行gdb命令

    • b *地址：在指定地址设置断点

    • 0x4010ae：add bl, dh gadget 的地址

  3. 计算

    • -ex "r" ：运行程序

    • \$rdx：RDX 寄存器（需要转义 $）

    • >> 8：右移8位（获取高8位）

    • & 0xFF：取低8位

运行结果：

可以看到计算出 $1 (即dh) = 32 (0x20)。或者直接看下面RDX显示的0xfbad208b

(注：

这是 _IO_FILE 结构的标志位。0xfbad 是 glibc 的魔法数字，标识这是一个 _IO_FILE 结构；而标志位 0x208b，

  • 0x2000 = _IO_USER_BUF (0x2000) - 用户提供的缓冲区

  • 0x0080 = _IO_CURRENTLY_PUTTING (0x0080) - 当前正在写入

  • 0x0008 = _IO_NO_READS (0x0008) - 不允许读取

  • 0x0002 = _IO_LINE_BUF (0x0002) - 行缓冲

  • 0x0001 = _IO_UNBUFFERED (0x0001) - 无缓冲)

所以，DH 值是固定的：来自 stdin 的 flags 字段，每次都是 0x20，无法改变，除非能修改内存中的 stdin 结构本身

step2

现在我们确认了，每次 add bl, dh 会给 bl 加 0x20，所以需要加上0x20的倍数，加法后的地址必须是另一个有用函数的地址

许哥的游戏

一、程序分析

  1. checksec发现需要绕过canary

  2. alarm+5里有syscall

    注意：libc-2.27.so特有

  3. Weapon

    ==>set_zero，任意地址写0

  4. Fight（close(1)和close(2)关闭标准输出和标准错误

    ==>challenge3，存在溢出

二、调试过程

（一）challenge3确认偏移

在pwndbg中，cyclic 240（反编译中看到缓冲区248），进入challenge3后输入，Ctrl C中断

stack 50查看栈布局

可以看到rbp前就是canary，所以偏移：248+canary+rbp+return_addr

（二）size_zero写 tls 中 canary 为0

i）利用原理

  1. 任意地址写 0：
    • str 是一个全局变量（地址固定）
    • lVar1 是用户输入的有符号长整数
    • 最终写入地址：str + lVar1
    • 写入内容：8 字节的 0（undefined8 表示 8 字节）
  2. 目标：tls 中的 canary：
    • 在 Linux x64 中，栈保护 canary 存储在 TLS（Thread Local Storage）中
    • 位置：fs:[0x28]（或 gs:[0x28]，取决于编译）
    • 每次函数开始时从 fs:[0x28] 加载到栈上
    • 函数结束前检查栈上的值是否与 fs:[0x28] 中的一致

ii）寻找 tls canary 地址

  1. 首先在pwndbg中 info registers

    看到 fs 基址为0x794eabc8c580

  2. 再 x/gx 查看0x28偏移处，通过 stack 20 确认到这里确实是 canary 的地方

  3. 因为要加上str指针，所以再查看一下str处的数据

    所以我们需要的输入：0x794eabc8c580 + 0x28 - 0x794eabc8a000 = 0x25a8

（三）bless第二关改 alarm 为 syscall

这里有任意地址加5，刚好匹配 alarm + 5 = syscall，可以改got表。

bless后验证一下：

三、漏洞利用

最后，在栈溢出部分，有 rbp 的 off-by-null，可以将rbp的最后一个字节覆盖为0，刚好是前面缓冲区22个ret的地址，而且由于执行到了challenge3和Fight的末尾，有两个leave，ret刚好可以栈迁移到缓冲区，执行ROP链

一、思路

从反编译可以看到，只要secret_key等于0x1337c0de就能获得flag。题目中dprintf有明显格式化字符串漏洞，用户通过read函数可以控制dprintf的第二个参数。

二、步骤

step1

首先验证漏洞，并找到输入的偏移（%p 是 printf 系列函数的格式说明符，用于以十六进制形式打印指针地址。）

从回显可以看到我们的输入aaaaaaaa（0x6161616161616161）在第5个参数处。

这里说明另两个格式化字符：

  1. %n 是 printf 系列函数中唯一具有写入能力的格式说明符，它可以将已输出的字符数写入指定的内存地址。

    格式符 写入大小 目标类型
    %n 4字节 int *
    %hn 2字节 short *
    %hhn 1字节 char *
    %lln 8字节 long long *

  2. %c 用于输出单个字符，将对应的参数解释为 unsigned char 并输出对应的ASCII字符。%{value}c则会输出value个字符

结合这两个工具，我们可以构造形如%{value_high}c%10$n达到任意地址写。
