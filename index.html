二、解决方案

泄露 libc

基于以上信息，我们形成以下思路：

  1. 索引一个40字节的句子(选项2)（因为40字节刚好是后面节点的大小，我们让两者大小相同才可通过 fastbin 复用）。
  2. 删除句子(选项1)。
  3. 此时我们在 fastbin 中存在一个40字节的空闲块(旧缓冲区)，但所有旧节点依然存在，所有 sentence_ptr 依然指向空闲块，也就是旧的句子缓冲区。
  4. 索引一个比原句子大16字节以上的新句子（我们需要让新节点malloc(0x28)复用 fastbin 中已释放内存，而不是句子缓冲区复用内存，这里为了防止内部碎片管理机制把已释放内存分配给句子缓冲区，索引一个比原句子大16字节以上的新句子更稳妥）。
  5. 此时分配了一个新句子缓冲区，而 fastbin 中的空闲块被复用，作为新的节点。注意这里原链表依然存在，新节点会继续连接到旧链表。虽然新节点的 sentence_ptr 字段会更新为新缓冲区地址，但是旧节点的 sentence_ptr 依然指向旧缓冲区，也就是此时的新节点。
  6. 查找旧节点中的单词，然后删除句子。我们现在在fastbin中有一个40字节的空闲块，而这个块仍然被用作节点，这是第二次释放此处内存。
  7. 索引一个40字节的句子，再次复用此处内存，控制节点内容，写入 got 表项。
  8. 当我们搜索该节点，伪造指针指向的内容会被打印出来，这会导致内存泄漏。

当要释放一个 fastbin 大小的堆块 p 时，系统会先找到对应的 fastbin 链表，并用 old 变量记录下当前位于链表头部的第一个空闲块，之后执行一个检查 old == p，也就是判断正在释放的堆块 (p) 是否和链表最前面的空闲块 (old) 是同一个；如果 old == p 条件成立，意味着用户试图连续两次释放同一个堆块，此时程序就会因 “double free or corruption (fasttop)” 错误而终止。

以上便是 Double Free 技术，那么这个技术到底实现了什么呢？下面详细探究：

复用 fastbin 中的空闲块时，glibc 堆管理器会把 main_arena 指向的堆块进行分配，然后将 main_arena 更新为这个堆的 fd 指针。

当你调用 index 创建一个新句子时，程序会 malloc 一块内存，并把你的输入写入这块内存。如果我们能控制 malloc 返回的地址是 __free_hook，那么我们输入的句子内容就会直接覆盖掉 __free_hook 的值。Fastbin Double Free 的作用，就是欺骗 glibc，让它误以为 libc 数据段上的某个地址是一个空闲的 fastbin 块。逻辑链条是这样的：

  1. Double Free：我们让同一块 40 字节的堆内存被释放两次。这会让 fastbin 链表形成一个环（chunk A -> chunk B -> chunk A ->...）。

  2. 链表劫持：因为链表里有重复的地址，我们可以通过第一次 malloc 拿到 chunk A，向 chunk A 里写入数据。由于 chunk A 同时也是链表的第三个节点，我们写入的数据就覆盖了 fastbin 链表的 fd 指针。

  3. 指向任意地址：我们把 fd 指针修改为 __free_hook 附近的地址。这样，glibc 就会认为那个地址是下一个空闲块。

  4. 分配到目标：第二次 malloc(40) 返回 chunk B。第三次 malloc(40) 返回 chunk A。 第四次 malloc(40)就会返回我们伪造的那个地址（也就是 __free_hook 附近）。

  5. hook 劫持：此时输入的句子内容就会直接覆盖掉 __free_hook 的值。

另外，由于ASRL，这里数据的高位在0x70~0x7f之间变化，而标准的 Fastbin 大小只有 0x20, 0x30, 0x40, 0x50, 0x60, 0x70, 0x80。没有 0x7a 或 0x7b 这种大小。

那 0x7a 为什么能用来欺骗系统呢？这其实涉及到 安全检查的精度的问题：

  • 分配时的校验逻辑：当 glibc 从 Fastbin 取出一个 chunk 时，它会计算 chunksize()，并通过 fastbin_index() 宏来确定这个大小属于哪个 bin。关键在于，这个索引的计算是右移 4 位 (即除以 16) 然后减 2 。
  • 计算结果是宽泛的区间：无论是 0x78、0x79、0x7a… 一直到 0x7f，右移 4 位后的结果都与 0x70 相同。
    • 公式：idx = (sz >> 4) - 2
    • 代入 0x7d：(0x7d >> 4) - 2 = 7 - 2 = 5（对应 0x70 的 bin）
    • 代入 0x70：(0x70 >> 4) - 2 = 7 - 2 = 5
  • 所以，对于负责“验票”的 fastbin_index() 宏来说，0x7a 和 0x70 指向的是同一个入口。因此，即使伪造的 size 是 0x7a，glibc 也会把它当作 0x70 这个 bin 里的合法成员，从而让检查通过。这就是为什么字节错位伪造的 0x7a 能够绕过硬件的 size 校验。

需要注意，不要使用__malloc_hook地址上方8字节数据去错位，因为 glibc 会认为 fastbin 返回的指针是堆的元数据，之后会向用户区读入数据(+0x10偏移)，如果太靠近__malloc_hook，__malloc_hook会有一部分作为size字段，另一部分作为用户区，会导致 one-gadget 地址被截断。通常来说使用远一点的偏移再通过填充覆写__malloc_hook比较可靠。

unlink 堆块

回顾一下 unlink 操作

FD是高地址堆块，BK是低地址堆块，注意fd在堆头偏移0x10处，bk在堆头偏移0x18处，P-> fd 实际上指 P+0x10 处的值，P-> bk 实际上指 P+0x18 处的值。也就是说，FD是fd指向的堆块，BK是bk指向的堆块。

理解两个写入操作，FD->bk实际是 FD+0x18，BK->fd实际是 BK+0x10，两个写入操作就是把BK写入到 FD 的0x10偏移处，把FD写入到 BK 的0x18偏移处。

同理，我们伪造 fd 和 bk，设置 fd = target_addr - 0x18，bk = content，于是FD = target_addr - 0x18，BK = content，然后执行写入操作，target_addr = content，content + 0x10 = target_addr - 0x18。

注意检查，要求 FD->bk == P（target_addr == P），BK->fd == P（content + 0x10 == P）。

二、调试

i）prev_size 域的复用

首先我们需要了解 prev_size 域的复用。我们知道堆块间保持16字节对齐，一般来说我们申请堆块的实际大小是类似0x20这样以0结尾的值。但是如果我们多申请了8字节，ptmalloc 应该如何保持16字节对齐呢？

实际上，对于 prev_size 域，前一个堆块在使用时(PREV_INUSE = 1)，它就没什么用了，这块8字节的区域就是空置的(8字节的0)。当前一个堆块在16字节对齐的基础上多申请8字节，此时 ptmalloc 会将这里未使用的8字节 prev_size 域复用，当作上一个堆块多申请的8字节内存

调试查看一下，如下申请两个堆块

可以看到，申请24字节的堆块，实际上是16字节的块加上8字节的prev_size域。这个特性正好促成了本题的off-by-one利用，溢出的一字节可以覆盖 size 域的低字节 (也就是这里的0x21)。

ii）fastbin 特性

glibc 2.23 尚未引入 tcache，对于一个大小介于 0x20~0x80 且不紧邻 top chunk的堆块(包含元数据)，在被 free 后会直接进入 fastbin。

fastbin 称为”单向链表”，即通过 fd 指针前向链接。我认为更精确的说法应该是”多条单向链表”，因为 free 不同大小的堆块会各自形成链表。

理解这里需要了解 fastbin 管理的逻辑。要知道，fastbin 链表在没有 free 堆块之前是不存在的，但 fastbinsY 数组是预先存在的。fastbinsY 是一个有10个格子的数组，位于 malloc_state 结构体中 。每个格子对应一种特定大小的空闲块（例如，第一个格子管理 0x20 字节的块，第二个管理 0x30 字节的块，以此类推）

当调用 free 释放一个大小在 fastbin 范围内（比如 0x20 字节）的堆块时，free 函数会计算出这个块应该属于哪个 fastbin（比如 0x20 对应的格子），它会去查看 fastbinsY 数组中对应格子的当前值，然后，它会把要释放块的 fd 指针设置为这个“头部”的地址，最后，更新 fastbinsY 中对应格子的值，让它指向刚刚释放的这个块。以此类推就形成了多条单向链表，每条链表对应一个大小。

理解了上面就可以明白 重用已释放内存 的逻辑

很明显，这里从 fastbin 取用内存并不会判断这片内存的位置以及与其他堆块的关系，我们依照下图说明整个漏洞利用思路。

iii）思路

大体来说我们的目标是泄露 libc 基址找到 system 地址，并且通过 content 写入 /bin/sh，然后覆写 free@got 为 system 地址，最后通过 free 触发 system(“/bin/sh”)。

首先我们申请两个 heap，每个 heap 程序会 malloc 两个堆块(申请时向 heap0 读入/bin/sh)，如上图所示。我们第0个堆块的 content 部分申请大小是 0x18，根据 prev_size域复用，实际上分配了一个0x20的堆块(包含元数据)，于是off-by-one会溢出到下一个堆块的 size 域。content1大小是0x20，所以修改 size域为 0x41，形成堆块重叠。

delete_heap 的两个free会分别释放 content 堆块和 struct 堆块，因为size域不同，两者会挂载到两个fastbin单向链表。

之后我们再申请一个0x30的 heap，也就是0x20的 struct 以及0x40的 content。这里申请的两个堆块大小和之前free的相同，所以重用已释放内存。此时堆块重叠，content块可以覆盖struct块，于是我们输入content内容把struct块里的content指针覆盖为 free@got，再通过show(1)打印这个new heap即可泄露 free 真实地址，从而推算libc基址。

因为我们把content指针覆盖成了 free@got，所以我们在edit_heap时是向 got 表读入数据，于是我们可以通过 edit 把 free@got 覆写为 system 地址，之后在调用 free 时就是调用 system。

最后，我们delete(0)调用system，因为向heap0开头读入了/bin/sh，所以实际上执行了system(“/bin/sh”)

对于此攻击方法，最主要的防御措施就是Full RELRO，如果程序编译时开启了 gcc -z now，整个 GOT 表在程序启动并完成所有函数地址解析后，会被标记为只读，任何尝试修改 GOT 表的操作都会导致程序崩溃。

ScrabASM

一、题目分析

查看程序，发现程序分配了一块固定地址 0x13370000 的可读可写可执行区域，还会执行这里的机器码。第一反应是构造 14 字节的 shellcode，但调试看了一下执行0x13370000时的寄存器状态根本无法构造出一个 14 字节的 shellcode，这是第一个难点。

再看源码，发现程序会调用 rand() 来控制 0x13370000 区域的机器码，虽然可以通过swap再次通过 rand() 生成一个字节，但程序只会输出一次字节序列，就是我们无法知道swap的字节是什么，这是第二个难点。

看到rand()后，又出现一个想法，就是rand()生成的是伪随机数，由种子决定，能否通过覆盖种子来控制伪随机数生成？但是看遍了源码都没有找到任何有机会溢出的漏洞点。

二、漏洞

隔了一天再看这道题，既然rand()依靠种子生成伪随机数，而种子通过当前的时间戳生成，srand(time(NULL))以秒为单位，所有只要在同一秒的时间去使用 srand(time(NULL))，生成的种子就相同，说明我们可以利用这一点在本地生成rand()伪随机数，以此推算出程序生成的伪随机数，甚至推算出需要swap多少次可以生成需要的数字。

让ai写个脚本验证一下

在 seed = int(time.time()) 那里，由于远程可能会有延迟，所以远程时需要 -1 秒。

一、调试

在学习SROP攻击之前，我们需要了解一下当 向程序发出信号 且 程序有相应的信号处理程序 时会发生什么

下面给出一个简单的程序，以便观察 signal 信号。使用以下命令编译代码gcc signal.c -o signal

当我们按下Ctrl + C，TTY 驱动检测到 INTR 字符 (^C)，内核生成 SIGINT，并发给“前台进程组”。本质上讲，Ctrl+C 默认会“退出程序”，因为 SIGINT 的默认行为 = terminate，相当于 kill(getpid(), SIGINT)。当然，这里signal(SIGINT, handler)接收到信号后会跳转执行注册的 handler，作为一个 临时插入执行的函数 。

我们在gdb里打开这个程序，由于gdb也是前台进程组的一员，Ctrl+C不会发给调试的程序，所以在run前需要执行handle SIGINT nostop noprint pass，以便让程序接到signal。之后b handler，r，Ctrl+C

选项 意思
nostop GDB 不因 SIGINT 停下
noprint GDB 不打印 signal 信息
pass 把 SIGINT 传给被调试程序

注意看上图 handler 的 ret 指令，handler 作为临时插入执行的函数，handler的 ret 理应返回到main，但是这里却返回到了 __restore_rt ，非常奇怪。

其实，在 SIGINT 到来之后（handler 之前），内核已经完成了：

  1. 在用户栈上压入 signal frame（并非sub rsp，而是内核改 regs->rsp）
  2. 在 frame 里保存所有寄存器
  3. 把 RIP 改成 handler
  4. 把“返回地址”设为 __restore_rt
  5. 切到用户态执行 handler

这里 __restore_rt 的唯一功能就是触发 rt_sigreture 系统调用，让内核用当前栈上的 signal frame 恢复寄存器。我尝试si步入观察syscall之后的指令，但是执行完这条后会直接进入sleep，原因是rt_sigreturn不会返回到 syscall 的下一条指令，它是一步之内“原子级完成”的，所以可以说：

rt_sigreturn 的“参数”是整个栈

另外需要注意这里mov rax , 0xf，syscall,我们SROP使用的就是这里的系统调用号0xf(15)，来调用SYS_rt_sigreturn。

下面探究一下整个 signal 机制，对SROP攻击不是很重要，仅用于加深理解。

平时C语言写的signal(SIGINT, handler) ，起始只是对 rt_sigaction 的一层“老接口封装”，这个过程可以称为”注册”，就是告诉内核当接收到SIGINT时，去执行handler，而非SIGINT 默认的处理方式 kill

之前讲到：

“当我们按下Ctrl + C，TTY 驱动检测到 INTR 字符 (^C)，内核生成 SIGINT，并发给“前台进程组”

更精确讲，内核生成 SIGINT 就是把 SIGINT 对应的 bit 置进目标进程（/线程）的 pending 集合里。这一步叫做 signal pending（信号挂起），此时只是有一个标记，不会让内核立即执行压入 signal frame 等操作。

之后内核会在某个安全点（比如从 syscall 返回用户态）检查：有没有 pending signal；有没有被屏蔽；有没有注册 handler。这一步叫做 signal delivery（信号传递），完成上面压入 signal frame 那5步。

总结一下

rt_sigaction 注册 signal 规则，告诉内核遇到SIGINT要去handler；
Ctrl+C 产生 SIGINT → 标记 pending；
内核在安全点 delivery 信号，构造 rt_sigframe，跳转 handler；
handler 返回后通过 rt_sigreturn 恢复寄存器。

SROP 本质上就是： 伪造 pending 已经被处理之后的那一帧。
