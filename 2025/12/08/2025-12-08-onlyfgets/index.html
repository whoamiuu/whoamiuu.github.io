CTF题解：onlyfgets

Posted on 2025-12-08 Edited on 2025-12-14 In Writeup

onlyfgets

一、程序分析

程序只有NX保护，且有简单栈溢出。但是，查看程序没有system等可以直接获得shell的gadget

有signal和alarm，根据提示gadget：

注：

endbr64（End Branch 64-bit）是 Intel CET（Control-flow Enforcement Technology）技术的一部分，用于防止ROP攻击。

作用：endbr64 ; 标记这是一个合法的间接跳转目标

工作原理：

  1. 运行时检查：当程序通过间接跳转（如 call rax, jmp [rbx]）执行时
  2. 如果目标地址不是 endbr64 指令 → 触发异常
  3. 如果是 endbr64 指令→ 正常执行

分析一下两个原语

假设我们可以控制 rbp 和 ebx，就可以向任意地址 (rbp - 0x3d) 加任意值 (ebx)

二、调试过程

（一） 原语利用

step1

根据上述分析，我们尝试控制 rbp 和 ebx，查一下 pop 指令：

可以看到，我们有了gadget

  • 0x000000000040114d : pop rbp ; ret
  • 0x00000000004011fc : pop rdi ; ret

但是无法直接控制 ebx。思考，ebx 可以通过 bl + dh 修改，通过调试命令查看一下dh

注：

  1. 输入

    • -e：启用转义字符解释

    • AAAAAAAA...：40个 ‘A’（32缓存，8 rbp）

    • \xae\x10\x40\x00\x00\x00\x00\x00：小端序的 0x4010ae（add bl, dh gadget）

  2. 断点：

    • -ex：执行gdb命令

    • b *地址：在指定地址设置断点

    • 0x4010ae：add bl, dh gadget 的地址

  3. 计算

    • -ex "r" ：运行程序

    • \$rdx：RDX 寄存器（需要转义 $）

    • >> 8：右移8位（获取高8位）

    • & 0xFF：取低8位

运行结果：

可以看到计算出 $1 (即dh) = 32 (0x20)。或者直接看下面RDX显示的0xfbad208b

(注：

这是 _IO_FILE 结构的标志位。0xfbad 是 glibc 的魔法数字，标识这是一个 _IO_FILE 结构；而标志位 0x208b，

  • 0x2000 = _IO_USER_BUF (0x2000) - 用户提供的缓冲区

  • 0x0080 = _IO_CURRENTLY_PUTTING (0x0080) - 当前正在写入

  • 0x0008 = _IO_NO_READS (0x0008) - 不允许读取

  • 0x0002 = _IO_LINE_BUF (0x0002) - 行缓冲

  • 0x0001 = _IO_UNBUFFERED (0x0001) - 无缓冲)

所以，DH 值是固定的：来自 stdin 的 flags 字段，每次都是 0x20，无法改变，除非能修改内存中的 stdin 结构本身

step2

现在我们确认了，每次 add bl, dh 会给 bl 加 0x20，所以需要加上0x20的倍数，加法后的地址必须是另一个有用函数的地址
